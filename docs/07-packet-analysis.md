# Packet Analysis

---

# Objective

This document explains how packet capture was used to observe and verify network communication within the pfSense home lab.

Rather than relying solely on configuration, packet analysis provided direct evidence of how traffic traversed the firewall, how Network Address Translation (NAT) modified packets, and how different network protocols behaved.

---

# Why Packet Capture?

Packet capture allows administrators to observe real network traffic.

It answers questions such as:

- Is the packet reaching the firewall?
- Which protocol is being used?
- Was the packet translated by NAT?
- Is the traffic leaving the correct interface?
- Is the firewall blocking or allowing the packet?

Packet analysis is one of the most valuable troubleshooting techniques in networking and cybersecurity.

---

# Packet Capture Configuration

Packet captures were performed using the pfSense Packet Capture utility.

| Setting | Value |
|---------|-------|
| Capture Tool | pfSense Packet Capture |
| Interfaces | LAN, WAN |
| Protocols | ICMP, TCP, UDP |
| Purpose | Traffic Verification |

Packet captures were taken on both interfaces to compare traffic before and after processing by the firewall.

---

# ICMP Analysis

ICMP traffic was generated using:

```bash
ping 8.8.8.8
```

## LAN Capture

Observed:

| Source | Destination |
|---------|-------------|
|192.168.10.100|8.8.8.8|

The packet left Ubuntu using its private IP address.

---

## WAN Capture

Observed:

| Source | Destination |
|---------|-------------|
|10.0.2.15|8.8.8.8|

The packet source changed to the WAN address after Source NAT.

---

# DNS Analysis

DNS requests were generated by:

```bash
ping google.com
```

Observed:

- UDP Port 53
- DNS query
- DNS response
- Successful hostname resolution

This confirmed that DNS forwarding was functioning correctly.

---

# HTTP Analysis

HTTP requests were generated using:

```bash
curl http://example.com
```

Observed:

- TCP
- Destination Port 80

When the HTTP block rule was enabled, traffic was denied according to the configured firewall policy.

---

# HTTPS Analysis

HTTPS requests were generated using:

```bash
curl https://example.com
```

Observed:

- TCP
- Destination Port 443

HTTPS traffic successfully passed through the firewall.

---

# Before and After NAT

The most significant observation during packet analysis was Source NAT.

## Before NAT

| Interface | Source |
|-----------|--------|
|LAN|192.168.10.100|

---

## After NAT

| Interface | Source |
|-----------|--------|
|WAN|10.0.2.15|

The packet destination remained unchanged.

Only the source address was translated.

# Key Takeaways

Packet analysis transformed firewall configuration into observable network behavior.

Instead of assuming that NAT, routing, and firewall rules operated correctly, packet captures provided direct evidence of each stage of packet processing.

Understanding packet captures is an essential skill for firewall administration, incident response, and network troubleshooting.
